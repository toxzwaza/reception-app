# 02. データベース設計

> 接続: 全モデルがデフォルト接続（`akioka_db`）を使用。文字コードは `utf8mb4_unicode_ci`。
> 出典: `database/migrations/` 全29本の最終形（後続マイグレーションの add/remove/変更を反映）＋ `reception_app.sql`（2026-04-16ダンプ）。

## 重要な前提（必読）

### A. `akioka_db` は外部システムと共有

`users` / `groups` / `initial_orders` / `stocks` は本アプリのマイグレーションに**存在しない**（別システム＝生産管理系が管理）。本アプリはこれらを論理参照する（DB上の外部キー制約は張らない）。

### B. 「担当者」の二重表現に注意

`staff_member_id` というカラム名でも、テーブルにより参照先が異なる。

| テーブル | `staff_member_id` の参照先 | FK制約 |
|---------|--------------------------|--------|
| `visitors` | **`users.id`** | なし（インデックスのみ） |
| `appointments` | **`users.id`** | なし（インデックスのみ） |
| `deliveries` | **`staff_members.id`** | あり |
| `pickups` | **`staff_members.id`** | あり |

`visitors` / `appointments` は当初 `staff_members` をFK参照していたが、`users` 参照へ変更後にFK制約を削除した経緯がある（後述の変更履歴参照）。

### C. マイグレーション vs SQLダンプの差異

`reception_app.sql` は受付系コアテーブルのみを含み、**2025-11-06以降のスケジュール／プロジェクト系6テーブル**（`facilities` / `schedule_events` / `schedule_participants` / `user_schedules` / `project_groups` / `project_group_user`）の CREATE TABLE が含まれない。本番 `akioka_db` には適用済みの想定だが、ダンプ取得元との不一致に注意。
さらに Outlookカラム追加マイグレーション（`2026_04_16_000001`）はダンプの `migrations` 記録に無く、ダンプ時点で未適用。

## ER関連図（テキスト）

```
users（外部管理） ──< user_schedules            (1:N)
users           ──1 groups（belongsTo / 外部管理）
users           >──< project_groups             (N:M via project_group_user)
users           ──< visitors      (staff_member_id)
users           ──< appointments  (staff_member_id)
users           ──< notification_recipients (user_id)
users           >──< schedule_events           (N:M via schedule_participants)
users           ──1 staff_members (1ユーザー=1スタッフ)

staff_members   ──< deliveries    (staff_member_id, FKあり)
staff_members   ──< pickups       (staff_member_id, FKあり)
staff_members   ──< visitors      (※モデル上 hasMany あり。実DBは users 参照)

facilities      ──< schedule_events             (1:N)
deliveries      >──< initial_orders（外部管理）   (N:M via delivery_initial_order)

notification_settings ──< notification_recipients (1:N, ON DELETE CASCADE)
```

## テーブル定義一覧

### 1. `staff_members` — スタッフ⇔ユーザー連携

`users` をラップした「受付担当者」エンティティ。当初は氏名・メール・部署・Teams情報・電子印を保持していたが、`users` へ集約され `user_id` のみに簡素化された。

| カラム | 型 | NULL | デフォルト | 用途 |
|--------|-----|------|-----------|------|
| id | bigint UNSIGNED (PK) | NO | AI | 主キー |
| user_id | bigint UNSIGNED | YES | NULL | `users.id`参照（FK制約なし） |
| created_at / updated_at | timestamp | YES | NULL | 作成・更新 |

**変更履歴**: ①初期は `name`/`email`(unique)/`department`/`teams_id`/`electronic_seal` を保持 → ②`teams_webhook_url` 追加 → ③`user_id` 追加 → ④`simplify`で氏名・メール・部署・Teams・電子印を**全削除**し `user_id` のみに。
**モデル `StaffMember`**: `$fillable = [user_id]`、`belongsTo User` / `hasMany Visitor` / `hasMany Delivery` / `hasMany Pickup`。
