# 05. 認証とセキュリティ

本アプリには**3層の認証/認可**が混在している。実運用の管理画面ガードは (A) の独自実装。

---

## (A) localStorage 認証（管理画面の実運用認証）

独自実装。フロント `resources/js/utils/auth.js` ＋ ミドルウェア `CheckLocalStorageAuth`（エイリアス `localstorage.auth`、`app/Http/Kernel.php` 登録）＋ `LocalStorageAuthController`。

### フロント（`auth.js`）

- `localStorage` に `user_id` と `user_data`(JSON) を保存。
- `login(userId, password)`: `/auth/login-local` にPOST → 成功時 `setUserId`/`setUserData`。
- `verifyUserId(userId)`: `/auth/verify` で検証。
- `logout()`: `/auth/logout-local` POST後 `clearAuth()`。
- `isAuthenticated()`: `user_id` の存在のみで判定。CSRFトークンはmetaタグから付与。

### サーバ（`CheckLocalStorageAuth`）

- user_id を **クエリパラメータ → セッション(`localStorage_user_id`) → ヘッダ`X-User-ID`** の順で解決。
- クエリで渡されたuser_idはセッションに保存（URLにuser_idを付けて遷移する想定）。
- user_id無し → JSONは401＋`redirect_to`、それ以外はloginへリダイレクト。
- 取得したuser_idで `User::where('del_flg',0)` を検索し、存在すれば `setUserResolver()` でrequestにセット。
- `/admin/*` 全ルートに適用。

### サーバ（`LocalStorageAuthController`）

| メソッド | ルート | 処理 |
|---------|--------|------|
| `login` | POST `/api/login-local` | `User`(id, del_flg=0)取得→`$input === $user->password`（**平文比較**）→`session(['localStorage_user_id'=>id])`→JSON。**平文をログ出力** |
| `logout` | POST `/api/logout-local` | クライアント側クリア前提 |
| `setSessionUser` | POST `/api/set-session-user` | セッションにuser_id保存 |
| `testPassword` | POST `/api/test-password` | **デバッグ用。平文パスワードを応答に露出**（del_flg未フィルタ） |

## (B) Laravel Sanctum（APIトークン認証）

- `User` が `HasApiTokens` を使用。
- `routes/api.php` の `/api/user` のみ `auth:sanctum` で保護。
- 導入済みだが活用範囲は限定的。

---

## (C) Laravel Breeze 標準セッション認証（`auth.php`）

- `/login`・`/register`・`/logout`・パスワードリセット等の標準フロー。`auth`/`guest` ミドルウェア。
- `Profile/Edit.vue` が `AuthenticatedLayout` を使用。
- ただし管理画面の実ガードは (A) であり、Breezeのセッション認証は管理画面では未使用（両者併存・役割分離）。**登録/リセットは `Hash::make`** を使う一方、(A) のログインは平文比較で不整合。

## 認証マトリクス

| 領域 | 認証方式 | 備考 |
|------|---------|------|
| 受付タブレット（`/`, `/visitor/*`, `/appointment`, `/interview`, `/delivery`, `/pickup` 等） | なし | タッチ端末で誰でも操作する前提 |
| 管理画面（`/admin/*`） | (A) localStorage認証 | 平文・admin判定なし |
| `/api/user` | (B) Sanctum | |
| `/login`・`/register`・`/profile`・`/password` | (C) Breeze | 管理画面ガードには未連携 |

## 既知のセキュリティリスク（改修候補）

> ⚠️ 以下は実コードに基づく指摘。追加開発時の優先検討事項。

1. **パスワードが平文保存・平文比較**
   `LocalStorageAuthController@login` は `$validated['password'] === $user->password`（Hash不使用）。`User` モデルのコメントにも「ハッシュ化されていない」と明記。`akioka_db` の `users.password` が平文である外部システム仕様に依存している。

2. **管理画面の認可が user_id の存在チェックのみ**
   `is_admin` を検証していない。`del_flg=0` の任意ユーザーが user_id を localStorage / クエリ / `X-User-ID` ヘッダに入れれば `/admin/*` にアクセス可能。**管理者ガードが効いていない。**

3. **デバッグ用エンドポイントの残存**
   `/api/test-password` が本番ルートに残り、平文パスワードを応答で返す。`/test`（`TestController@test`）も `dd()` でデータをダンプ。

4. **平文パスワードのログ出力**
   `login` で入力パスワードをログに記録している。

5. **フロントと実ルートのパス不整合**
   `auth.js` の `/auth/*` と実ルート `/api/*` が不一致（`verify` は未登録）。動作不全・想定外挙動の原因になりうる。

6. **外部HTTP連携でSSL検証無効**
   - Teams通知: `Http::withOptions(['verify'=>false])`
   - `TwilioTestController`: SSL検証無効（開発用と明記）

7. **受付系APIが認証なし**
   `/api/initial-orders`・`/api/com-names` は発注データ（業者名等）を認証なしで返す。社内ネットワーク前提と思われるが要確認。

---

## 改修時の推奨事項（参考）

- 管理画面ミドルウェアに `is_admin = 1` チェックを追加。
- `/api/test-password`・`/test` ルートを本番から除去（または環境ガード）。
- パスワードのログ出力を削除。
- `auth.js` のfetch先を実ルート（`/api/*`）に統一、または `/auth/*` ルートを定義。
- 外部連携のSSL検証を本番で有効化。
- 受付系APIに最低限のトークン/IP制限を検討。
